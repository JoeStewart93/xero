# Xero

Xero is a modular Command & Control (C2) platform for **authorized cybersecurity research**, defensive security testing, and scoped red-team operations.

> **Authorized use only.** Deploy beacons, handlers, scanners, and pivot routes only on systems and networks where you have explicit written permission. See [Authorized Use](spec/mvp-requirements.md#10-authorized-use).

## Architecture At A Glance

Xero runs as separate Docker Compose stacks:

- **Local UI/BFF stack:** frontend, local FastAPI BFF, PostgreSQL, and Redis. This is the operator console and local login surface.
- **C2 backend stack:** separate FastAPI C2 backend, PostgreSQL, and Redis. This can run locally for development or on a remote host later.
- **Beacon handler stack:** external handler scaffold with health/readiness plus optional C2 pairing/heartbeat; handler tunnel behavior remains planned.
- **Scanner stack:** external scanner scaffold with health/readiness plus optional C2 pairing/heartbeat; scanner execution remains planned.

The operator logs in to the local UI/BFF first, then connects the UI to a C2 backend from Settings.

Default local URLs:

| Service | URL |
| :--- | :--- |
| Frontend | `http://localhost:3000` |
| Local BFF API | `http://localhost:8000` |
| C2 backend API | `http://localhost:8001` |
| Beacon handler scaffold | `http://localhost:8002` |
| Scanner scaffold | `http://localhost:8003` |

Default development credentials:

| Account | Username | Password |
| :--- | :--- | :--- |
| Operator | `operator` | `operator_password` |
| Local admin | `admin` | `admin` |
| C2 connection | n/a | `c2_password` |

Change all defaults before using any non-local environment.

- Node.js and npm

## Install And Run Locally

Clone the repository and enter the platform directory:

```powershell
git clone <repo-url> xero
cd xero\platform
```

The stack works with built-in Compose defaults. Copy `.env.example` only when you want to customize ports, passwords, or C2 settings:

```powershell
Copy-Item .env.example .env
```

Start the local UI/BFF stack:

```powershell
docker compose -f docker-compose.bff.yml up -d --build
```

`docker-compose.yml` is kept as a temporary compatibility alias for the BFF stack, but new commands should use `docker-compose.bff.yml`.

Start the local C2 backend stack:

```powershell
docker compose -f docker-compose.c2.yml up -d --build
```

Check container health:

```powershell
docker compose -f docker-compose.bff.yml ps
docker compose -f docker-compose.c2.yml ps
```

Optional service scaffolds can be started separately:

```powershell
docker compose -f docker-compose.handler.yml up -d --build
docker compose -f docker-compose.scanner.yml up -d --build
```

When starting a scaffold manually with C2 pairing, provide the values generated by `Settings > Infrastructure`:

```powershell
$env:C2_BASE_URL="http://localhost:8001"
$env:WORKER_PAIRING_TOKEN="<generated-token>"
$env:WORKER_NAME="edge-scanner-01"
docker compose -f docker-compose.scanner.yml up -d --build
```

Open the UI:

```text
http://localhost:3000
```

Log in with `admin/admin` or `operator/operator_password`.

Connect the UI to the local C2 backend:

1. Open `Settings`.
2. Set Backend URL to `http://localhost:8001`.
3. Set C2 password to `c2_password`.
4. Click `Connect`.

After connection, the C2 indicator should show connected and the Beacons page can display registered/heartbeating beacons.

To manage infrastructure workers, open `Settings` and then the `Infrastructure` subtab. From there you can:

- View the embedded C2 beacon handler and embedded C2 scanner.
- Generate one-time pairing commands for external handler/scanner nodes.
- Launch or stop local C2-managed handler/scanner scaffold containers when `C2_LOCAL_PROVISIONING_ENABLED=true`.

The handler/scanner containers store their worker session under their `/data` volume and continue heartbeating to C2 after the first successful pairing.

For protocol validation work, enable the development harness with `C2_PROTOCOL_FRAME_HARNESS_ENABLED=true`. The harness exposes `GET /api/v1/protocol`, `POST /api/v1/protocol/frames`, and `GET /api/v1/security/events` for the F0011 binary frame contract.

The primary live beacon transport is `ws://<c2-host>/ws/beacon` with subprotocol `xero.beacon.v1`. It carries encrypted F0011 binary frames, returns encrypted ACKs, tracks active connections through C2-token-protected `GET /api/v1/transport`, and surfaces transport state in `Settings > Infrastructure` and the Beacons overview/detail views. Authenticated HTTP long-poll fallback is available through beacon-token-protected poll/frame endpoints; real task queues and full task-result storage remain later feature work.

## Port Conflicts

If `3000`, `8000`, or `8001` are already in use, set alternate ports before starting the stacks.

Example:

```powershell
$env:FRONTEND_PORT="13000"
$env:BACKEND_PORT="18000"
$env:C2_BACKEND_PORT="18001"
$env:FRONTEND_ORIGIN="http://localhost:13000"
$env:VITE_DEFAULT_C2_BASE_URL="http://localhost:18001"

docker compose -f docker-compose.bff.yml up -d --build
docker compose -f docker-compose.c2.yml up -d --build
```

Then open:

```text
http://localhost:13000
```

When using alternate ports, connect Settings to the matching C2 URL, such as `http://localhost:18001`.

The Docker frontend uses a same-origin nginx proxy for local BFF `/auth` and `/api` calls, so it does not need `VITE_API_BASE_URL` when `BACKEND_PORT` changes. Set `VITE_API_BASE_URL` only when serving the frontend outside the Docker compose frontend container.

`FRONTEND_ORIGIN` must match the browser origin exactly for direct browser requests to the BFF API.

## Smoke Test The Stack

Basic health checks:

```powershell
Invoke-WebRequest http://localhost:8000/health -UseBasicParsing
Invoke-WebRequest http://localhost:8000/ready -UseBasicParsing
Invoke-WebRequest http://localhost:8001/health -UseBasicParsing
Invoke-WebRequest http://localhost:8001/ready -UseBasicParsing
```

Authenticate to C2:

```powershell
$body = @{ password = "c2_password" } | ConvertTo-Json
Invoke-RestMethod -Uri http://localhost:8001/api/v1/c2/connect -Method POST -ContentType "application/json" -Body $body
```

Register a development beacon record:

```powershell
$beacon = @{
  machine_fingerprint_hash = "local-dev-fingerprint-001"
  hostname = "local-dev-host"
  os = "Windows 11"
  architecture = "x64"
  internal_ip = "10.10.0.10"
  external_ip = "198.51.100.10"
  pid = 4242
} | ConvertTo-Json

$registration = Invoke-RestMethod -Uri http://localhost:8001/api/v1/beacons/register -Method POST -ContentType "application/json" -Body $beacon
$registration
```

Send a heartbeat with the returned one-time beacon token:

```powershell
$headers = @{ Authorization = "Bearer $($registration.beacon_token)" }
Invoke-RestMethod -Uri "http://localhost:8001/api/v1/beacons/$($registration.beacon_id)/heartbeat" -Method POST -Headers $headers -ContentType "application/json" -Body "{}"
```

The Beacons page should show the host as online with a recent heartbeat.

Create and register a development scanner worker:

```powershell
$c2 = Invoke-RestMethod -Uri http://localhost:8001/api/v1/c2/connect -Method POST -ContentType "application/json" -Body (@{ password = "c2_password" } | ConvertTo-Json)
$auth = @{ Authorization = "Bearer $($c2.access_token)" }
$pairing = Invoke-RestMethod -Uri http://localhost:8001/api/v1/infrastructure/pairing-tokens -Method POST -Headers $auth -ContentType "application/json" -Body (@{ kind = "scanner"; name = "local-dev-scanner" } | ConvertTo-Json)
$workerBody = @{
  kind = "scanner"
  name = "local-dev-scanner"
  pairing_token = $pairing.pairing_token
  endpoint = "http://scanner.local:8000"
  capabilities = @("tcp-connect")
  capacity = 4
  current_load = 0
  version = "dev"
} | ConvertTo-Json
$worker = Invoke-RestMethod -Uri http://localhost:8001/api/v1/infrastructure/workers/register -Method POST -ContentType "application/json" -Body $workerBody
$heartbeatBody = @{ capabilities = @("tcp-connect"); capacity = 4; current_load = 1; version = "dev" } | ConvertTo-Json
Invoke-RestMethod -Uri "http://localhost:8001/api/v1/infrastructure/workers/$($worker.worker_id)/heartbeat" -Method POST -Headers @{ Authorization = "Bearer $($worker.worker_token)" } -ContentType "application/json" -Body $heartbeatBody
```

## Run Tests

From `platform/`, use the project virtual environment if it exists:

```powershell
.\.venv\Scripts\python.exe scripts\ci.py backend-lint
.\.venv\Scripts\python.exe scripts\ci.py backend-unit
.\.venv\Scripts\python.exe scripts\ci.py backend-behave
.\.venv\Scripts\python.exe scripts\ci.py openapi-check
.\.venv\Scripts\python.exe scripts\ci.py go-protocol-test
npm --prefix frontend run lint
npm --prefix frontend test -- --run
npm --prefix frontend run build
```

For integration and Playwright tests, keep both Docker stacks running:

```powershell
.\.venv\Scripts\python.exe scripts\ci.py backend-integration
npm --prefix frontend run test:e2e
```

If you use alternate ports, provide the matching test URLs:

```powershell
$env:XERO_BACKEND_URL="http://localhost:18000"
$env:XERO_C2_BACKEND_URL="http://localhost:18001"
$env:PLAYWRIGHT_BASE_URL="http://localhost:13000"
$env:PLAYWRIGHT_C2_BASE_URL="http://localhost:18001"
npm --prefix frontend run test:e2e
```

## Stop Or Reset

Stop the local UI/BFF stack:

```powershell
docker compose -f docker-compose.bff.yml down
```

Stop the C2 stack:

```powershell
docker compose -f docker-compose.c2.yml down
```

Stop optional handler/scanner scaffolds:

```powershell
docker compose -f docker-compose.handler.yml down
docker compose -f docker-compose.scanner.yml down
```

C2-managed worker launches use separate Docker Compose project names with the `xero-managed` prefix by default. They can be stopped from `Settings > Infrastructure`; if needed, inspect them with `docker ps` or `docker compose ls`.

Remove persistent PostgreSQL and Redis volumes when you want a clean local database:

```powershell
docker compose -f docker-compose.bff.yml down -v
docker compose -f docker-compose.c2.yml down -v
```

## Troubleshooting

- **Port is already allocated:** set alternate `FRONTEND_PORT`, `BACKEND_PORT`, and/or `C2_BACKEND_PORT`, then rebuild.
- **Login or Settings connection fails in the browser:** verify `FRONTEND_ORIGIN` matches the exact UI URL.
- **C2 shows disconnected:** confirm the C2 stack is healthy and use the correct backend URL/password in Settings.
- **OpenAPI check fails after API work:** run `.\.venv\Scripts\python.exe scripts\ci.py openapi-export`, inspect the service spec diff, then rerun `openapi-check`.
- **Need clean default users:** stop the relevant stack with `down -v`, then start it again to reseed development users.
